# 无人机模块实现审计报告（ruoair-uav / ruoair-ui）

## 1. 审计范围与依据

- **后端范围**：`ruoair-uav`（controller/service/mapper/domain）
- **前端范围**：`ruoair-ui/src/views/uav`、`ruoair-ui/src/api/uav`、`ruoair-ui/src/views/index.vue`
- **需求依据**：`doc/无人机需求书.md`

说明：本报告重点审计无人机业务模块（设备、航线、任务、结果、首页大盘）。系统管理（用户/角色/日志）主要依赖若依框架既有能力，仅做关联性判断，不做逐页面深度代码核验。

---

## 2. 总体结论（先给结论）

- **已实现较完整**：设备/航线/任务/结果四大模块的基础 CRUD、任务状态流转、地图绘制与轨迹回显、任务完成自动生成结果、首页统计大盘。
- **部分实现或存在偏差**：任务取消语义、筛选维度、结果快照完整性、结果导出字段、部分约束校验与后端兜底、UI一致性与易用性。
- **未实现/明显缺口**：AI识别框取（拓展项）、任务状态变更记录、结果模块按需求的多维筛选与“巡防时长/完成时间”等关键字段闭环。

---

## 3. 需求对照：已实现 / 未实现

## 3.1 系统管理模块（3.1）

- **已实现（框架层）**：登录、RBAC、用户与日志能力在若依基础模块中具备。
- **本次模块内未新增实现**：`ruoair-uav` 与 `views/uav` 未扩展系统管理特性（这点是合理的分层）。

---

## 3.2 设备管理（3.2）

### 已实现

1. 设备 CRUD、分页、导出、按编号/名称/型号/状态筛选（`SysUavEquipmentController`、`SysUavEquipmentMapper.xml`、`views/uav/equipment/index.vue`）。
2. 设备编号唯一校验（新增时）。
3. 删除前校验是否绑定任务（`SysUavEquipmentServiceImpl`）。
4. 任务开始/结束时设备状态联动（`SysUavTaskServiceImpl.startTask/completeTask/cancelTask`）。

### 未实现或与需求不一致

1. **字段不完整**：需求里的“摄像头参数、归属人”未在设备领域模型和页面实现（`SysUavEquipment` 仅有 code/name/model/flightDuration/status/remark）。
2. **设备详情页与历史任务记录**未实现（当前只有列表+弹窗编辑）。
3. **状态枚举语义存在偏差风险**：需求提“正常/维修中/已报废/离线/任务中（模拟）”，代码使用字典值 `0/1/2/3`，但业务注释处存在“离线/故障/任务中”表述，需统一词典与业务文案。
4. **更新唯一性校验有缺陷**：编辑时按编号查到自身也会判重，可能导致“原编号不变也无法保存”（`updateSysUavEquipment` 的判空逻辑未排除当前ID）。

---

## 3.3 航线管理（3.3）

### 已实现

1. 高德地图绘制航线、坐标存储（`routePoints` JSON）。
2. 地图回显航线和点位序号。
3. 地点搜索与定位、清除重画。
4. 航线 CRUD、分页、导出、按名称筛选。

### 未实现或与需求不一致

1. **删除约束缺失**：需求要求“已绑定任务航线不可删”，当前后端未做绑定校验（`SysUavRouteServiceImpl.delete...` 直接删）。
2. **航线关联历史任务记录视图**未实现。
3. 航线编辑更偏“重画”，缺少细粒度点位编辑体验（可优化，不算硬缺陷）。

---

## 3.4 任务管理（3.4）

### 已实现

1. 任务创建校验：设备可用、航线未被未完成任务占用、开始时间不可早于当前时间（后端已兜底）。
2. 状态机接口：开始、完成、取消（`/uav/task/start|complete|cancel`）。
3. 状态联动：开始->设备任务中；完成/取消->设备恢复正常。
4. 完成任务自动生成结果快照（任务名/设备名/航线名/执行人/概述）。
5. 前端任务列表有动态操作按钮、进度条、监控地图模拟（轨迹、剩余时间、进度）。

### 未实现或与需求不一致

1. **任务基础信息缺字段**：需求有“任务描述”，当前任务模型/页面无该字段。
2. **任务列表展示维度不达标**：需求希望展示设备/航线信息，当前主要展示 `equipmentId/routeId`，缺少设备名、航线名直观展示。
3. **筛选维度不完整**：需求提“任务名称、执行人、设备名称、状态、时间范围”，当前后端主要是任务名/设备ID/航线ID/状态/开始时间范围，缺执行人与设备名称维度。
4. **任务详情与状态变更记录**未实现。
5. **执行中“取消”语义被前端改成“暂停模拟”**，与需求“取消=状态改已取消且不生成结果”有冲突风险（后端支持取消，但当前主交互更偏“暂停/继续”）。

---

## 3.5 结果管理（3.5）

### 已实现

1. 完成任务自动生成结果记录。
2. 结果 CRUD、分页、导出。
3. 结果页面可编辑概述/发现/处理/备注，支持图片URL预览。
4. 结果页面可回看航线地图（通过 task->route 反查）。

### 未实现或与需求不一致

1. **结果字段不完整**：需求有“巡防时长、完成时间”等展示与导出要素，当前领域模型和页面未形成闭环字段。
2. **筛选维度不足**：需求希望按任务名称、设备、执行人、时间范围筛选，当前后端查询仅 `resultCode/taskId`。
3. **快照抗删不彻底**：结果地图回显依赖实时任务和航线；若任务或航线删除，结果页可能无法回显轨迹。  
   （需求强调快照与归档，理想方案应把必要轨迹数据也快照到结果侧）
4. **导出字段不完全符合需求**：需求强调包含“发现情况”等，`SysUavResult` 中 `findings` 未加 `@Excel` 注解，导出不全。
5. **AI识别框取**（3.5.3拓展）未实现，仅保留图片URL字段。
6. **后端编辑校验较弱**：需求要求“巡航概述必填”，当前主要是前端规则，后端服务未强校验。

---

## 3.6 首页大盘与界面需求（第4章）

### 已实现

1. 首页四项总数统计（设备/航线/任务/结果）。
2. 设备状态分布饼图、近7天任务趋势折线图。

### 未实现或可改进点

1. 饼图状态名称当前取数据库 `status` 值，缺少友好中文映射（代码注释也提到了这一点）。
2. 大盘接口未加权限注解（按系统安全策略可补）。

---

## 4. 前端UI可优化建议（重点）

以下为按优先级排序的可落地优化项。

## P0（优先立即做）

1. **任务、结果、设备页“ID化交互”改造**  
   当前大量用 `equipmentId/routeId/taskId` 输入框，建议改为下拉选择（显示名称+ID），避免误填，提升演示观感。

2. **监控看板信息层级整理**  
   统一“进度显示规则（整数）+进度条文本策略+状态文案（开始/暂停/继续/取消）”，避免语义冲突和视觉重复。

3. **状态颜色与词典统一**  
   统一后端状态含义、字典项、前端显示标签；避免“离线/故障/维修中”混用。

## P1（中优先）

1. **详情抽屉/详情页补齐**  
   设备详情（含历史任务）、航线详情（含关联任务）、任务详情（含状态流转时间线）、结果详情（含快照字段）。

2. **列表字段优化**  
   任务列表增加设备名/航线名，结果列表增加完成时间/巡防时长，提升业务可读性。

3. **筛选器增强**  
   结果页补任务名/设备名/执行人/时间范围；任务页补执行人和设备名称筛选。

4. **统一地图组件封装**  
   当前任务/航线/结果页面都有地图逻辑，建议抽成公共组件（加载、异常、绘制、回显、动画统一）。

## P2（体验与工程化）

1. AMap Key 改为环境变量，避免硬编码。
2. 表单数字字段改 `el-input-number` + 单位后缀（分钟、米）。
3. 长列表列宽、固定列、`show-overflow-tooltip` 统一。
4. 操作按钮危险动作分层（二次确认+说明影响）。
5. 页面空状态、加载状态、错误提示风格统一。

---

## 5. 后续修复建议（后端优先级）

1. 修复设备编辑判重逻辑（排除当前ID）。
2. 补航线删除绑定校验（有未完成任务引用则拒删）。
3. 结果查询与导出字段补齐（含 findings、完成时间、巡防时长、执行人等）。
4. 结果快照补轨迹数据，解除对 task/route 存量数据的依赖。
5. 任务取消/暂停语义拆分：  
   - 取消（业务状态终止）  
   - 暂停（仅模拟层，不改变业务终态）  
   避免与需求冲突。

---

## 6. 审计结论

项目已完成无人机业务闭环的**主干路径**（设备→航线→任务→结果）并具备可演示性，但距离需求书“完整验收”仍有若干关键差距，主要集中在：

1. 结果模块字段与筛选闭环不足；
2. 航线/设备删除约束与详情能力不足；
3. 任务取消语义与当前“暂停模拟”交互存在冲突；
4. UI在业务可读性与一致性上仍可显著提升。

建议优先完成 P0/P1 项，以保证答辩与验收场景的稳定性和说服力。
